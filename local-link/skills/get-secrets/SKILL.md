## Workflow

1. 从用户输入获取阅读资料，如用户输入为空则从会话上下文获取待处理资料（或知识）
2. 基于用户指定的阅读资料（或知识），输出 **4 条**进阶知识
3. 为每条知识各进行**极其简单的验证**（最多 2 轮 open-websearch 工具调用）
4. 由你按照置信度（权重 0.3）和深度（权重 0.7）两者综合排序，把结果按模板展示给用户

要求：
1. **绝不修改**用户给出的原文档
2. 不要包含对原文档的“审判”或“引用”，专注于知识
  2.1 最坏的情况你把原文档的错误以“常见认知错误”间接引导读者
3. 验证过程必须非常简单，每个知识不超过 2 次工具调用
4. 每条知识的输出严格遵循以下格式（仅知识即可，无需输出“验证得分”等杂项）

## 输出格式

```
<num>. <一句话brief，控制在15字以内>

<随知识复杂度逐层深入的详细描述>
```

描述要求：
- 用自然叙述，不要列表式结构（不要"背景：xxx 原理：xxx"）
- 一句话能讲清不用两句话，但信息密度要足够
- 随复杂度自然展开，从现象到机制到影响

## 质量要求

每条知识必须满足以下至少一项：
- **踩坑型**：真实工程中会遇到，文档不会告诉你，但一踩一个准
- **反直觉型**：和表面认知相反，能展示对底层机制的理解
- **工程杠杆型**：掌握后可以直接优化 latency / 成本 / 准确率
- **区分度型**：说出去能让人从"背八股"切换到"这人懂系统"

## 禁止事项

- 禁止编造第一人称"我们在xxx项目中遇到"之类的虚假经历
- 禁止输出文档里已经明确写了的基础概念
- 禁止输出泛泛而谈的最佳实践（如"要注意异常处理"）
- 禁止输出没有具体案例或根因分析的空洞观点
